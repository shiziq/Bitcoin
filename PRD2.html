Section 1Product Vision

We are integrating USDT stablecoin capabilities into the KCB banking application. Users load value from their bank accounts, send it peer-to-peer, and pay merchants at point-of-sale — all on Polygon + Arbitrum blockchain networks with flat fees near KSh 0.15 and 2-second settlement. Every monetary value displays simultaneously in USDT and Kenyan Shillings.

M-Pesa — Percentage Fee Model
  • 1–3% per transaction — KSh 100–300 on KSh 10,000
  • SMS-based 2FA vulnerable to SIM-swap attacks
  • SMS payment trail — transaction history on phone records
  • Non-interoperable across telcos (Safaricom-only)
  • No stablecoin — KES only, exposed to inflation
Bato — Flat Fee Model
  • KSh 0.15 flat blockchain fee — same for KSh 500 or KSh 50,000
  • In-app authenticator + device key — zero SMS, zero SIM-swap risk
  • All confirmations in-app — zero SMS trail, full privacy
  • Cross-telco, cross-chain — Polygon + Arbitrum, Solana + TRON in Phase 2
  • Always shows KSh equivalent — users budget in the currency they know

Section 2The Seven Principles

Every design decision, every code review, and every QA test must pass these seven gates. These are the constitutional foundation of the product.

Principle 1

KES is the Primary Reference

Kenyan users budget, price, and think in shillings. Every screen, receipt, and notification must present the KES amount prominently. The KES value drives user comprehension and trust.

Principle 2

USDT is the Settlement Currency

All on-chain transfers, smart contracts, and value storage use USDT. USDT provides stability against KES fluctuation and enables fast, cheap blockchain transactions.

Principle 3

Every Amount Shows Both

Balance: USDT + KES. Transfers: USDT + KES. Fees: USDT + KES. History: USDT + KES. Notifications: USDT + KES. Receipts: USDT + KES. No exception. This is the golden rule of the entire system.

Principle 4

Rates Are Transparent

Show the rate source (CBK, Binance P2P), the bank spread percentage, and the rate freshness indicator on every conversion. Users deserve to know where the numbers come from.

Principle 5

Historical Rates Are Immutable

January's receipt shows January's rate — never today's. Transaction records store the exact KES/USDT rate at the time of transaction. The rate column is immutable after confirmation.

Principle 6

No SMS. Period.

Authentication: in-app authenticator + device key. Confirmation: in-app push. Recovery: in-app KYC. Zero SMS trail. Zero email. All payment records exist only inside the bank app.

Principle 7

Built for Kenya, By Design

KES formatting: en-KE locale. Timezone: Africa/Nairobi. Rate feeds: Kenyan sources first. Phone format: +254. Every design choice serves the Kenyan banking customer.

Section 3 — Build This FirstUser Registration

One-time registration using the user's Kenyan phone number. No email. No SMS. The phone number links to the existing KCB customer profile. A device-bound key pair is generated in the Secure Enclave/TEE — this becomes the permanent authentication credential.

Section 4 — Build This SecondAuthentication — In-App Authenticator

The device key bound during registration authenticates every transaction. No SMS. No email. No biometric OS prompt. The authenticator is an in-app component — a push-based confirmation dialog inside the bank app. Two verification modes: Mode A (Tap to Approve) for standard transactions, Mode B (Shared Code Matching) for high-value or high-risk transactions.

Mode A — Tap to Approve

Standard transactions. Authenticator shows dual-currency payment preview. User taps Approve. Authenticator signs challenge with device key from Secure Enclave. Backend verifies signature.

Mode B — Shared Code

High-value transactions. Backend generates 6-char alpha-numeric code displayed on BOTH the POS terminal and the Authenticator. User verifies codes match, enters code. 60s TTL, max 3 attempts.

Section 6Rate Engine & Dual Currency Display

Three independent Kenyan market feeds (CBK, Binance P2P, Forex) feed a weighted average engine. A configurable bank spread is applied. The result is cached for 15 seconds and distributed to every display component via the DualCurrencyDisplay client-side SDK.

Currency Conversion Engine: Rate Feeds → Weighted Avg → Spread → 15s Cache → UI

Section 7Dual Currency Display — The Golden Rule

⛔
Every screen, notification, receipt, and confirmation MUST show both USDT and KES. This is a non-negotiable system-wide invariant. Balance, transfer, fees, history, notifications, receipts — no exceptions.

Display Rules

Primary: USDT amount. Secondary: KES approximate with ≈ prefix. Rate line: current conversion rate. Freshness indicator: Live ● / Recent ○ / Stale ⚠. Auto-refresh every 15 seconds.

KES Rounding

Locale: en-KE (1,500,000.00). 2 fraction digits. Half-up rounding. Grouping separator: comma. Prefix: "KSh ". Micro-amounts display as cents.

Rate Transparency

Source disclosed on every screen. Rate lock state: Estimated (orange) → Locked (green). Timestamp on all rates for audit. Spread shown as percentage.

Freshness Indicators

● Live (<30s) · ○ Recent (<2min) · ⚠ Stale (>2min). Refresh on app foreground and every 15 seconds in background.

Section 10Withdrawal — Cash Out USDT to Bank

Users convert USDT from their digital wallet back into Kenyan Shillings deposited to their linked KCB bank account. USDT → KES using the current sell rate with the bank's 2% spread. Confirmation requires the in-app authenticator.

Withdrawal Flow

User taps Cash Out → enters USDT amount → sees KES receivable → confirms via Authenticator → backend calls bridgeBurn() on smart contract → KCB core banking credits savings account. Dual-currency receipt stored in-app. No SMS trail.

Section 16Database Schema

User Registrations & Devices 
CREATE TABLE user_registrations (
    user_id UUID PRIMARY KEY, phone_number VARCHAR(15) UNIQUE NOT NULL,
    phone_number_hash VARCHAR(64) NOT NULL, wallet_address VARCHAR(42) NOT NULL,
    bank_account_id VARCHAR(50) NOT NULL, status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE TABLE user_devices (
    id UUID PRIMARY KEY, user_id UUID REFERENCES user_registrations(user_id),
    device_id VARCHAR(255) NOT NULL, public_key TEXT NOT NULL,
    platform VARCHAR(20) NOT NULL, is_current BOOLEAN DEFAULT TRUE,
    UNIQUE (user_id, device_id)
);
Transactions, Exchange Rates & Receipts 
CREATE TABLE transactions (
    id UUID PRIMARY KEY, user_id UUID NOT NULL, type VARCHAR(50) NOT NULL,
    amount_usdt DECIMAL(18,6) NOT NULL, amount_kes DECIMAL(18,2) NOT NULL,
    fee_usdt DECIMAL(18,6), fee_kes DECIMAL(18,2),
    rate_usdt_to_kes DECIMAL(18,6) NOT NULL, rate_source VARCHAR(255),
    rate_spread_multiplier DECIMAL(5,4), rate_timestamp TIMESTAMP NOT NULL,
    blockchain_tx_hash VARCHAR(66), counterparty_type VARCHAR(20),
    counterparty_name VARCHAR(255), merchant_id VARCHAR(100),
    invoice_ref VARCHAR(100), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE TABLE exchange_rates (
    id SERIAL PRIMARY KEY, timestamp TIMESTAMP NOT NULL,
    base_rate DECIMAL(18,6) NOT NULL, effective_rate DECIMAL(18,6) NOT NULL,
    source_feeds TEXT[], is_stale BOOLEAN DEFAULT FALSE
);
CREATE TABLE digital_receipts (
    id UUID PRIMARY KEY, transaction_id UUID REFERENCES transactions(id),
    receipt_number VARCHAR(50) UNIQUE NOT NULL, receipt_data JSONB NOT NULL
);

Section 17API Endpoints — OpenAPI 3.0

POST/api/v1/auth/register

One-time registration with phone number. No SMS. Returns wallet address and device key confirmation.

GET/api/v1/rates/current

Current USDT/KES rate with source transparency, spread, and freshness indicator.

GET/api/v1/wallet/balance

Wallet balance with dual currency display — USDT raw + formatted, KES raw + formatted, rate info.

POST/api/v1/transfer/preview

Preview a P2P transfer with dual currency amounts, fees, and total cost before confirmation.

POST/api/v1/pos/scan

Process a scanned POS QR code. Returns payment preview with merchant details and shared code.

GET/api/v1/transactions

Paginated transaction history with dual currency display, historical rates, and period summaries.
