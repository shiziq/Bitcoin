Section 1

Product Vision — What We're Building & Why

We are integrating USDT stablecoin capabilities into an existing Kenyan banking application. Users load value from their bank accounts, send it peer-to-peer, and pay merchants at point-of-sale — all on Polygon blockchain with near-zero fees and 2-second confirmation. Every monetary value is displayed simultaneously in USDT and Kenyan Shillings.

The Problem

Today — M-Pesa Dominance
  • M-Pesa charges 1–3% per transaction — percentage-based, scales with amount
  • SMS-based 2FA is vulnerable to SIM-swap attacks — Kenya's most common fraud vector
  • SMS payment trails expose transaction history on phone records — no privacy
  • M-Pesa has single-digit finality on Safaricom network — can't interoperate with Airtel/Telkom seamlessly
  • Cross-border payments take days and cost 5–10% — remittance corridor is broken
  • Users cannot hold or spend stable USD-pegged value — KES-only, exposed to local inflation
  • No stablecoin integration in any Kenyan banking app — the gap is wide open
Tomorrow — Bato + KenyaPay
  • Blockchain TX fees near ~KSh 0.15 — FLAT FEE, not percentage-based
  • In-app authenticator + device-attestation — no SMS, no SIM-swap risk
  • Instant settlement on L2 networks (~2 second block time) — orders of magnitude faster
  • USDT stablecoin fully integrated into bank app — first-mover advantage
  • KES display on every value — users always know the shilling equivalent
  • All confirmations and receipts stored in-app — zero SMS trail, full privacy
  • Interoperable across all Kenyan telcos, merchants, and future cross-border corridors

The Competitive Edge — Why This Beats M-Pesa

M-Pesa charges a percentage-based fee — send KSh 10,000 and you pay KSh 100–300. Bato charges a flat blockchain fee of approximately KSh 0.15 per transaction — send KSh 10 or KSh 100,000 and the blockchain cost is the same. This is the fundamental economic advantage of stablecoin rails over mobile money.

Network Strategy — Multi-Chain for Maximum Competitive Advantage

To rival M-Pesa's dominance, Bato must deploy USDT across multiple blockchain networks — giving the bank and its customers the flexibility to choose the optimal network per transaction based on cost, speed, and destination. The executive decision is not "which chain" but "which chains and in what order."

The Seven Principles — Our Non-Negotiables

Every design decision, every code review, every QA test must pass these seven gates. These principles are the constitutional foundation of the product. Read them before writing a single line of code.

Principle 1

KES is the primary reference

Kenyan users budget, price, and think in shillings. Every user-facing screen, receipt, and notification must present the KES amount prominently. The KES value drives user comprehension and trust.

Principle 2

USDT is the settlement currency

All on-chain transfers, smart contract interactions, and value storage use USDT. USDT provides stability against KES fluctuation and enables fast, cheap, global blockchain transactions.

Principle 3

Every amount shows both

Balance: USDT + KES. Transfer: USDT + KES. Fees: USDT + KES. History: USDT + KES. Notifications: USDT + KES. Receipts: USDT + KES. No exception. No escape hatch. This is the golden rule of the entire system.

Principle 4

Rates are transparent

Display the rate source (CBK, Binance P2P, Forex), the bank spread percentage, and the rate freshness indicator on every screen showing converted values. Users deserve to know where every number comes from.

Principle 5

Historical rates are immutable

Transaction records and receipts store the exact KES/USDT rate at the time of transaction. January's receipt shows January's rate — never today's. The rate_usdt_to_kes column is immutable after confirmation.

Principle 6

No SMS. Period.

Authentication: in-app authenticator + device key. Confirmation: in-app notification center. Recovery: in-app KYC challenge. Zero SMS trail. Zero MNO dependency. Zero email. All payment records exist only inside the app.

Principle 7

Built for Kenya, by design

KES formatting: en-KE locale (1,500,000.00). Timezone: Africa/Nairobi (EAT). Rate feeds: Kenyan sources first. Phone format: +254 normalization. Every design choice serves the Kenyan banking customer.

User Registration — Phone Number, Once Only

Registration is a one-time event using the user's Kenyan phone number. No email verification. No SMS OTP. The phone number links to the existing bank customer profile. During registration, a device-bound key pair is generated in the Secure Enclave/TEE — this key becomes the user's permanent authentication credential.

class RegistrationService {
    async registerUser(request: RegistrationRequest): Promise<RegistrationResponse> {
        const normalized = this.normalizePhoneNumber(request.phoneNumber);
        const existing = await this.registrationDB.findByPhone(normalized);
        if (existing && existing.status !== 'closed') throw new PhoneAlreadyRegisteredError();
        const bankProfile = await this.coreBankingSystem.findCustomerByPhone(normalized);
        if (!bankProfile) throw new CustomerNotFoundError();
        const userId = crypto.randomUUID();
        const walletAddress = await this.walletService.createWallet(userId);
        const keyValid = await this.cryptoService.validateDeviceKey(request.devicePublicKey);
        if (!keyValid) throw new SecurityError('Invalid device key');
        await this.registrationDB.insert({ userId, phoneNumber: normalized, walletAddress,
            deviceKeys: [{ publicKey: request.devicePublicKey, ... }],
            bankAccountId: bankProfile.accountId, status: 'active' });
        return { userId, walletAddress, phoneNumberNormalized: normalized,
                 bankAccountLinked: { ...bankProfile }, deviceKeyConfirmed: true };
    }
}
Section 6 — Build This Second

Authentication — In-App Authenticator + Device Attestation

The device public key bound during registration is used for all subsequent authentication. After scanning a QR code or initiating a transfer, the user confirms the transaction using the in-app authenticator — a secure confirmation dialog built into the bank app itself. The server verifies the device signature from the Secure Enclave and the authenticator's signed challenge. Payment confirmations and receipts are stored exclusively in-app — there is no SMS trail whatsoever.

⚠
No biometric OS prompt. The authenticator is an in-app component — not the system FaceID/TouchID dialog. This keeps the payment confirmation experience fully contained within the bank app's UI.

Authenticator Message-Based Confirmation Flow

All payment confirmations flow through the Authenticator App via push messages — not embedded in the bank app, not via SMS. The backend sends a confirmation request to the Authenticator App on the user's registered device. The user receives the request, reviews it, and approves or rejects. Two verification modes are supported depending on risk level.

Authenticator App — Message-Based Confirmation Architecture with Two Verification Modes

Verification Mode A — Tap to Approve

For standard transactions where the risk score is below threshold. The Authenticator App displays the dual-currency payment preview and the user taps "Approve" or "Reject". The Authenticator signs a cryptographic challenge (nonce + transaction ID + amount) using the device-bound key from the Secure Enclave. The signature is sent back to the bank backend for verification.

Verification Mode B — Shared Code Matching

For high-value or high-risk transactions where merchant-side verification is required. The bank backend generates a 6-character alpha-numeric code (e.g., "A7K3M9") and delivers it simultaneously to both the merchant's POS terminal screen AND the user's Authenticator App. The user visually confirms the code matches on both screens, then enters the code in the Authenticator App. This prevents man-in-the-middle attacks — both parties verify the same code. The code has a 60-second TTL and a maximum of 3 entry attempts.

Rate Engine & Dual Currency Display SDK

This is the heart of dual-currency display. Three independent Kenyan market feeds feed a weighted average engine. A configurable bank spread is applied. The result is cached for 15 seconds and distributed to every display component via the DualCurrencyDisplay client-side SDK.

Currency Conversion Engine: Rate Feeds → Weighted Avg → Spread → Cache → UI
ℹ
Design Decision: We use 3 independent feeds with weighted averaging. If any feed fails, the engine falls back to remaining feeds. If all fail, it returns the last cached rate marked as stale. The system never blocks a transaction waiting for a rate.
Expand: KenyaRateEngine Full Implementation 
interface RateFeed { name: string; source: string; priority: number; weight: number; fetchRate: () => Promise<number>; }

class KenyaRateEngine {
    private feeds: RateFeed[] = [
        { name: 'CBK_REFERENCE', source: 'Central Bank of Kenya', priority: 1, weight: 0.4,
          fetchRate: async () => { /* fetch from api.centralbank.go.ke */ } },
        { name: 'BINANCE_P2P', source: 'Binance P2P Kenya', priority: 2, weight: 0.35,
          fetchRate: async () => { /* fetch USDTKES from Binance API */ } },
        { name: 'FOREX_AGGREGATOR', source: 'Aggregated Forex Feed', priority: 3, weight: 0.25,
          fetchRate: async () => this.forexProvider.getUSDTKESRate() }
    ];
    private cache = null;

    async getCurrentRate(): Promise<ConversionRate> {
        if (this.cache && (Date.now() - this.cache.timestamp) < this.cache.ttl)
            return this.buildRateResponse(this.cache.rate, this.cache.feeds);
        const rates = [];
        for (const feed of this.feeds) {
            try {
                const rate = await Promise.race([ feed.fetchRate(),
                    new Promise((_, reject) => setTimeout(() => reject(new Error('Timeout')), 3000)) ]);
                rates.push({ rate, feed: feed.source, weight: feed.weight });
            } catch (e) { console.warn(`Feed ${feed.name} failed`); }
        }
        if (rates.length === 0) { if (this.cache) return this.buildRateResponse(this.cache.rate, this.cache.feeds, true); throw new RateUnavailableError(); }
        const totalW = rates.reduce((s, r) => s + r.weight, 0);
        const weightedRate = rates.reduce((s, r) => s + r.rate * r.weight, 0) / totalW;
        const effectiveRate = weightedRate * 1.02;
        this.cache = { rate: effectiveRate, timestamp: Date.now(), feeds: rates.map(r => r.feed), ttl: 15000 };
        return this.buildRateResponse(effectiveRate, rates.map(r => r.feed));
    }
}

Section 9

Dual Currency Display — The Golden Rule

⛔
Every screen, notification, receipt, and confirmation that displays money MUST show both USDT and KES. This is a non-negotiable system-wide invariant. Balance, transfer amount, fees, history, notifications, receipts — no exceptions.

Display Rules

Every monetary value rendered in the system:

  • Primary line: USDT amount ("600.00 USDT")
  • Secondary line: KES approximate ("≈ KSh 87,000.00")
  • Rate line: current conversion rate ("1 USDT = KSh 145.00")
  • Freshness indicator: Live ● / Recent ○ / Stale ⚠

Rate Transparency

Users must always know where the rate comes from:

  • Rate source disclosed ("Source: CBK + 2% spread")
  • Rate lock state indicated (Estimated / Locked)
  • Pre-confirmation: orange "Estimated rate"
  • Post-confirmation: green "Locked rate"
  • Timestamp on all rates for audit trail

KES Rounding Specification

  • Locale: en-KE (1,500,000.00 format)
  • Min/max fraction digits: 2
  • Rounding: half-up
  • Grouping separator: comma
  • Decimal separator: period
  • Prefix: "KSh "
  • Micro-amounts (< KSh 1): display as cents

Freshness Indicators

  • ● Live — rate < 30 seconds old
  • ○ Recent — rate < 2 minutes old
  • ⚠ Stale — rate > 2 minutes old
  • Rate auto-refreshes every 15 seconds
  • Refresh triggered on app foreground

class POSPaymentService {
    async scanQRCode(qrString: string): Promise<POSPaymentPreview> {
        const payload = JSON.parse(qrString);
        if (payload.protocol !== "bankpay-ke-v1") throw new InvalidQRCodeError();
        const merchant = await this.merchantDirectory.getMerchant(payload.merchantId);
        const valid = await this.cryptoService.verifySignature(payload, payload.merchantSignature, merchant.signingPublicKey);
        if (!valid) throw new SecurityError('Invalid merchant signature');
        if (Date.now() > payload.expiresAt * 1000) throw new QRCodeExpiredError();
        const rate = await this.rateService.getCurrentRate();
        const usdtAmount = payload.amount.kes / rate.effectiveRate;
        const networkFee = await this.blockchainService.estimateTransferFee();
        const balance = await this.balanceService.getUserBalance(this.currentUserId);
        return { /* payment preview with dual currency display */ };
    }
    async executePOSPayment(preview, qrPayload, authProof) {
        if (Date.now() > preview.rate.expiresAt) { const fresh = await this.scanQRCode(JSON.stringify(qrPayload)); throw new RateExpiredError('Rate expired', fresh); }
        await this.authService.requireStrongAuth(authProof); // IN-APP AUTHENTICATOR
        const tx = await this.blockchainService.transferUSDT({ from: this.getUserAddress(authProof.userId), to: qrPayload.merchantAddress, amount: preview.payment.amountUSDT, memo: qrPayload.invoiceRef });
        // Record + notify + generate receipt — ALL IN-APP, NO SMS
    }

Section 14

PayBand — NFC Wearable Payment Bracelet (Hardware Vision)

As a forward-looking hardware component of the KenyaPay ecosystem, the PayBand is an NFC-enabled wearable bracelet that allows users to make contactless payments at merchant POS terminals by tapping the bracelet. The bracelet is loaded with USDT from the user's wallet via the bank app, and all payment confirmations flow through the Authenticator App on the user's phone. The bracelet uses the same device-key authentication model as the mobile app — a hardware secure element stores a factory-provisioned key that is bound to the user's bank account at pairing.

PayBand — NFC Wearable Bracelet: Load via App QR + NFC → Tap to Pay at POS → Authenticator Confirms

How It Works

Phase 1 — Load the Bracelet

1
User opens bank app, selects "Send to Bracelet"
Enters the amount in USDT to load. App displays a QR code encoded with the bracelet's band ID and the transfer amount.
2
User brings bracelet near phone NFC reader
The bracelet's NFC NTAG 424 DNA chip receives the NDEF payload containing the band ID, amount, and a signed transaction reference from the backend.
3
Authenticator push confirms the load
The Authenticator App receives a push notification: "Load 50.00 USDT (≈ KSh 7,250.00) to Bracelet #A3K9?" User taps Approve. Authenticator signs challenge with device key. Backend validates and credits the bracelet's on-chain balance.

Phase 2 — Pay with the Bracelet

1
Cashier rings up amount on POS, QR code + shared code appear
POS displays KSh 650 with QR code (bankpay-ke-v1) and the Mode B shared verification code (e.g., A7K3M9) on the merchant screen.
2
User taps bracelet to NFC reader on POS terminal
Bracelet beams NDEF payload: { bandId, nonce } via NFC. POS terminal captures the bracelet's identity and nonce.
3
Authenticator App receives confirmation request
Push notification: "Pay KSh 650.00 (≈ 4.48 USDT) to Naivas Supermarket — Moi Avenue? Shared code: A7K3M9." User visually verifies the code matches the POS screen, enters it in the Authenticator, and approves.
4
Backend verifies and executes transaction
Validates shared code + bracelet device key signature + user authenticator signature. Executes USDT transfer from bracelet's address to merchant's address on Polygon. POS confirms payment in <2 seconds.

Hardware Specifications

ComponentSpecification
NFC ChipNTAG 424 DNA — AES-128 authenticated NFC with Secure Unique NFC (SUN) message authentication
Secure ElementFactory-provisioned device key stored in tamper-resistant hardware. Keys never leave the chip.
DisplayLow-power E-ink display showing current USDT balance and shared verification code during transactions
BatteryCR2032 coin cell — 12-month estimated life under normal use (E-ink only draws power on refresh)
DurabilityIP67 water resistant. Silicone band. Operating temperature 0°C to 45°C.
PairingBound to user's bank account during initial NFC handshake. One bracelet per user account.
Data ProtocolNDEF (NFC Data Exchange Format) records containing bandId, encrypted payload, and device signature

Security Model

ℹ
Same trust model as the mobile app. The bracelet does not store funds locally — it is a signing device. The actual USDT balance lives on-chain at the bracelet's assigned wallet address. Every NFC transmission is signed with the bracelet's factory-provisioned device key. The backend verifies both the bracelet signature AND the user's authenticator confirmation before executing any transaction. If the bracelet is lost, the user can deactivate it from the bank app (requires authenticator confirmation). No private key ever leaves the secure element.
