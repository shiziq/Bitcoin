Section 1

Product Vision

We are integrating USDT stablecoin capabilities into the KCB banking application. Users load value from their bank accounts, send it peer-to-peer, and pay merchants at point-of-sale — all on Polygon + Arbitrum blockchain networks with flat fees near KSh 0.15 and 2-second settlement.

M-Pesa Today
  • 1–3% per transaction — KSh 100–300 on KSh 10,000
  • SMS-based 2FA vulnerable to SIM-swap
  • SMS payment trail on phone records
  • Non-interoperable across telcos
  • KES only — exposed to inflation
Bato Tomorrow
  • KSh 0.15 flat fee — same for KSh 500 or KSh 50,000
  • In-app authenticator + device key — zero SMS
  • All confirmations in-app — zero SMS trail
  • Cross-telco, cross-chain
  • Always shows KSh equivalent

Section 2

The Seven Principles

Every design decision, every code review, and every QA test must pass these seven gates. These are the constitutional foundation of the product.

Principle 1
KES is the Primary Reference
Kenyan users budget, price, and think in shillings. Every screen must present KES prominently.
Principle 2
USDT is the Settlement Currency
All on-chain transfers and value storage use USDT. Stability against KES fluctuation.
Principle 3
Every Amount Shows Both
Balance, transfers, fees, history, notifications, receipts — USDT + KES. No exception.
Principle 4
Rates Are Transparent
Show source (CBK, Binance P2P), spread %, and freshness indicator on every conversion.
Principle 5
Historical Rates Are Immutable
January's receipt shows January's rate — never today's. Rate column immutable after confirmation.
Principle 6
No SMS. Period.
Authentication: in-app authenticator + device key. Confirmation: in-app push. Zero SMS trail.
Principle 7
Built for Kenya, By Design
KES formatting: en-KE. Timezone: Africa/Nairobi. Rate feeds: Kenyan sources first. Phone: +254.

Section 4 — Build This Second

Authentication — In-App Authenticator

The device key bound during registration authenticates every transaction. No SMS. No email. No biometric OS prompt. Two verification modes: Mode A (Tap to Approve) for standard transactions, Mode B (Shared Code Matching) for high-value transactions.

Mode A — Tap to Approve

Standard transactions. Authenticator shows dual-currency payment preview. User taps Approve. Authenticator signs challenge with device key from Secure Enclave. Backend verifies signature.

Mode B — Shared Code

High-value transactions. Backend generates 6-char alpha-numeric code displayed on BOTH POS terminal and Authenticator. User verifies codes match, enters code. 60s TTL, max 3 attempts.

Section 6

Rate Engine & Dual Currency Display

Three independent Kenyan market feeds (CBK, Binance P2P, Forex) feed a weighted average engine. A configurable bank spread is applied. Cached for 15 seconds and distributed to every display component via the DualCurrencyDisplay client-side SDK.

Section 7

Dual Currency Display — The Golden Rule

Every screen, notification, receipt, and confirmation MUST show both USDT and KES. This is a non-negotiable system-wide invariant.
Display Rules

Primary: USDT amount. Secondary: KES with ≈ prefix. Rate line: current conversion. Freshness indicator: Live ● / Recent ○ / Stale ⚠.

KES Rounding

Locale: en-KE (1,500,000.00). 2 fraction digits. Half-up rounding. Grouping: comma. Prefix: "KSh ".

Rate Transparency

Source disclosed on every screen. Rate lock state: Estimated (orange) → Locked (green). Spread shown as percentage.

Freshness Indicators

● Live (<30s) · ○ Recent (<2min) · ⚠ Stale (>2min). Auto-refresh every 15s.
